#### 關鍵技術決策

**決策 1**: 為何選擇遺傳算法而非網格搜尋？

**選項**:
| 方案 | 優點 | 缺點 |
|------|------|------|
| 網格搜尋 | 保證覆蓋所有組合 | 17 參數需測試 10^17 組合，不可行 |
| 隨機搜尋 | 快速探索 | 無法利用歷史最佳結果 |
| 貝氏優化 | 高效收斂 | 實作複雜、對高維參數效果有限 |
| 遺傳算法 | 平衡探索與利用、並行計算 | 無法保證全局最優 |

**決策**: ✅ **遺傳算法**

**理由**:
1. **高維參數空間**: 17 個連續參數 + 1 個離散參數（調倉頻率）
2. **並行能力**: 族群 20-30 個體可並行評估（2-4 進程）
3. **探索與利用平衡**: 錦標賽選擇 + 精英保留 = 穩定收斂
4. **實證成功**: v1 達 54.18%，v2 達 74.51%，持續改進
5. **執行效率**: 25 代僅需 45-60 分鐘

## 三、重要開發決策紀錄

### 決策一：為何使用 Decimal128？

**問題：** 價格欄位應該使用 Float (number) 還是 Decimal128？

**決策：** ✅ **所有價格、金額欄位必須使用 Decimal128**

**理由：**
1. **精度要求：** Float64 在進行十進位小數運算時會產生不可預測的精度損失。例如 `0.1 + 0.2` 在電腦中不精確等於 `0.3`。
2. **複利效應：** 在進行長週期回測（5-10 年）時，微小誤差會被複利效應指數級放大，導致策略績效（如夏普比率、最大回撤）嚴重失真。
3. **業界標準：** 所有專業金融系統（Bloomberg、Refinitiv）中，凡是涉及「金額」的欄位，都使用定點數型別。
4. **法規要求：** 金融交易系統對精度有嚴格要求，使用 Float 可能導致法規合規問題。

**實施結果：**
- ✅ 5.12M 筆股價數據已全部使用 Decimal128
- ✅ 87 筆股利數據已全部使用 Decimal128
- ✅ 所有財報數據已全部使用 Decimal128

### 決策二：為何採用 P0 → P1 → P2 的三階段優化路徑？

**問題：** 是否應該同時進行數據修復和程式碼重構？

**決策：** ✅ **優先解決數據品質（P0），再優化架構（P1/P2）**

**理由：**
1. **Garbage In, Garbage Out：** 如果底層數據（股價、股利）是錯誤或不完整的，再優雅的程式碼也無法產出有意義的分析結果。
2. **風險控制：** 數據品質問題會直接導致錯誤的投資決策，是最高優先級的風險。
3. **價值優先：** 修復數據能立即提升系統可用性，而程式碼重構主要提升維護性。
4. **測試可靠性：** 只有在數據正確的基礎上，才能有效測試程式碼重構的結果。

**執行順序：**
- **P0（最高）:** 刪除 48,176 筆無效數據 → 邏輯正確率 99.9999%
- **P1（中等）:** 驗證 Schema 使用 Decimal128 → 精度 100%
- **P2（較低）:** 重構程式碼 → 重複率 43% → 0%

### 決策三：為何使用 FinMind API？

**問題：** 應該使用哪個數據來源？

**決策：** ✅ **使用 FinMind API 作為主要數據來源**

**理由：**
1. **數據廣度：** 提供 43 個資料表，涵蓋技術面、基本面、籌碼面
2. **易用性：** 標準 RESTful API，文檔清晰
3. **成本效益：** 免費方案已能滿足個人量化分析需求
4. **台灣專注：** 專門針對台股市場，數據品質較高
5. **社群支援：** 有活躍的開發者社群

**限制：**
- 免費版日配額：402 次/天
- 歷史數據：部分資料表從 2010 年開始

### 決策四：為何重構下載腳本為模組化架構？

**問題：** 8 個功能重複的下載腳本應該如何處理？

**決策：** ✅ **重構為統一的 src/downloaders/ 模組**

**理由：**
1. **DRY 原則：** Don't Repeat Yourself，避免重複代碼
2. **維護成本：** 原架構修改一個 API 邏輯需要同步 8 個檔案
3. **錯誤風險：** 重複代碼容易導致不一致和隱藏的 bug
4. **可擴展性：** 新架構便於添加新的數據來源和下載任務
5. **測試性：** 模組化架構更容易編寫單元測試

**重構結果：**
- **Before:** 8 個腳本，~3,200 行代碼，43% 重複率
- **After:** 5 個模組，~1,532 行代碼，0% 重複率
- **效益:** 代碼減少 52%，維護複雜度降低 70%

### 決策五：為何解釋三方審計差異？

  - 結論：需要重構程式碼架構
  
- **Copilot（動態查詢）:**
  - 方法：實際查詢 MongoDB 數據
  - 發現：數據品質問題（48,176 筆無效）
  - 結論：需要修復數據品質

**結論：** 兩者看的是不同層面（代碼 vs 數據），都是正確的，需要分別處理。

def batch_detect(self, stock_ids: List[str], date: str):
    results = []
    for stock_id in stock_ids:
        # 1. 從數據庫獲取歷史價格數據
        df = self._get_price_data(stock_id, date, lookback_days=120)
