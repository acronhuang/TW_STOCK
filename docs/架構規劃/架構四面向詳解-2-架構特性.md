# 架構四面向詳解 · 2 · 架構特性 (Architecture Characteristics)

> 架構特性＝驅動設計的隱性品質需求（-ilities）。評分 5=強力支撐…1=近乎缺席，反映實機觀察。

---

## 前三驅動特性（架構明顯為此而建）

### ⑤ 資料完整性 Data Integrity
整個系統圍繞「數字要對」而建：
- 價格/金額一律 `Decimal128`（避免浮點誤差）。
- T+1 回補邏輯（`backfill_recent_gaps.py`）補救官方 API 延遲。
- 冪等 upsert（去重鍵 `{stock_id,date}` / `{symbol,year,season}`）。
- 每日完整度檢查（`twse_openapi_sync --check-only`，13 表門檻）。
- 兩層復驗（`reverify_team.py`）：內部一致性 + FinMind 外部複核。
- 團隊分析內建查證層（`verify_metrics`，收盤 tol 3%、PE/殖利率 tol 5%）。

### ⑤ 成本 / 節儉 Frugality
零經常性雲端成本是明確前提：
- 全用免費官方 API（TWSE/TPEX OpenAPI 免 token、FinMind free/付費、台銀匯率免金鑰）。
- 自架 Ollama（.28/.27）取代付費 LLM API。
- 單機 + 兩台推理節點。

### ④ 可擴充性 Extensibility
- 高度模組化的 `src/`，新增分析法或選股策略不需動既有模組。
- 透過共享 DB 鬆耦合 → 任一分析法可獨立開發/替換。

### ④ 模組化 Modularity
- 依領域切分 bounded contexts；分析模組間不互相 import，只透過 MongoDB 交換資料。

## 支撐特性（有做到，非優先）

| 特性 | 評分 | 現況 |
|------|:--:|------|
| 可用性 Availability | ③ | API/Dashboard 為 systemd 自動重啟、開機自啟；但單節點、單 MongoDB、無 HA |
| 回應性 Responsiveness | ③ | API 讀取走 Mongo 索引快；重運算刻意批次（團隊 ~19h）；互動回測限 30 檔保即時 |
| 可恢復性 Recoverability | ③ | 週備份 MongoDB；分析逐檔存檔（中斷可續）；無 point-in-time recovery |
| 可觀測性 Observability | ② | 檔案日誌 + 監控頁 + 完整度檢查推 LINE；無 metrics/tracing |

---

## 犧牲特性（為換取前三驅動而放棄）

| 特性 | 評分 | 為何薄弱 |
|------|:--:|------|
| 水平擴展 Scalability | ② | 單 MongoDB、單主機；只能垂直擴展；LLM 靠加 Ollama 節點但編排器循序 |
| 安全性 Security | ① | API/Dashboard 無驗證；`.env` 明文密鑰；無 TLS；採「內網信任」 |
| 可部署性 Deployability | ① | 無 CI/CD；systemd+cron 手動；跨 OS 移轉痛點（時區/路徑/launchctl）即此特性薄弱的證據 |
| 可測試性 Testability | ② | 有 tests/ 但分析邏輯與 DB/LLM 深度耦合，難隔離單測 |

## 架構師判讀

這是一套 **「單一使用者、資料正確優先、零預算」** 的個人量化平台。工程資源全押在**資料可信**與**分析可擴充**上，理性地犧牲了多人系統才需要的擴展/安全/自動部署。

> 移轉到 Linux 時浮現的一連串問題（時區、路徑、指錯表），正是「可部署性」被犧牲的具體代價。詳見 [規劃-架構修正清單-實機對照](規劃-架構修正清單-實機對照.md)。

## 特性衝突與優先序（供決策用）

| 衝突對 | 本系統的選擇 |
|--------|-------------|
| 資料完整性 vs 回應性 | 完整性優先 → 重運算做成批次、可續跑 |
| 成本 vs 品質/速度 | 成本優先 → 開源模型自架，接受全市場 19h |
| 可擴充性 vs 可測試性 | 擴充性優先 → 鬆耦合但難單測 |
| 簡單性 vs 可部署性 | 簡單性優先 → 手動部署，移轉痛 |
