# 架構四面向詳解 · 3 · 架構決策 (Architecture Decisions / ADR)

> 從實機反推出的隱性 ADR。每個含：脈絡 → 決策 → 取捨。

---

## ADR-1 · 單一 MongoDB 作為整合中樞
- **脈絡**：多個獨立分析模組需交換價量財報資料。
- **決策**：全部經共享資料庫整合，不用訊息匯流排或服務 API。
- **取捨**：得到極簡整合與鬆耦合；失去 schema 契約保證，改欄位會靜默打斷下游（本次「指錯表 `financial_reports`」災情源於此）。

## ADR-2 · 自架 Ollama 多節點取代雲端 LLM
- **脈絡**：七角色團隊分析需大量 LLM 推理，全市場每日近 2000 檔。
- **決策**：.28/.27 自架 Ollama，用開源模型（qwen2.5 / hermes3）。
- **取捨**：零 API 成本、資料不出內網；換來自行維運 GPU、模型品質與速度受限（全市場 19h）。

## ADR-3 · .27/.28 節點分工做合議多樣性
- **脈絡**：單一模型判斷易有系統性偏誤。
- **決策**：價值/籌碼角色與合議委員放 .27（hermes3），主力推理放 .28（qwen2.5），刻意換視角。
- **取捨**：得到判斷多樣性與交叉驗證；換來雙節點協調複雜度與路由設定。

## ADR-4 · 兩階段團隊分析（quick / heavy）
- **脈絡**：顧問整合用大模型昂貴；六角色可平行且較輕。
- **決策**：phase1 先跑六角色存檔，phase2 再補顧問整合 + 合議。
- **取捨**：分離便宜平行工作與昂貴收斂工作、可分段續跑；換來兩階段狀態管理（檔名/日期一致性）複雜度。

## ADR-5 · 免費官方 API + T+1 回補
- **脈絡**：付費行情源成本高；官方 OpenAPI 免費但有 T+1 延遲與缺漏。
- **決策**：用免費源，配隔日回補 + 完整度檢查補救時效與缺口。
- **取捨**：零資料成本；換來資料時效延遲、冷門股/權證缺漏，需複雜回補與復驗機制。

## ADR-6 · 金額一律 Decimal128
- **脈絡**：股價/財務計算對精度敏感，浮點會累積誤差。
- **決策**：價格與金額欄位全存 `Decimal128`。
- **取捨**：資料完整性；換來每處讀取都要型別轉換（`safe_float`/`tof` 樣板）。

## ADR-7 · cron 批次而非事件驅動
- **脈絡**：台股資料本質是每日收盤結算，無需即時串流。
- **決策**：以 cron 排程批次驅動整條管線。
- **取捨**：極簡、易理解；換來時序脆弱——時區設錯即整條管線跑錯時間（本次移轉核心災情）。

## ADR-8 · 分析結果雙寫 JSON + DB
- **脈絡**：長時間分析需可中斷續跑，也需可查詢。
- **決策**：逐檔雙寫——JSON（續跑來源）+ `team_analysis` 集合（查詢來源）。
- **取捨**：兼得韌性與可查詢；換來兩份真相需保持一致（靠 migration 同步）。

## ADR-9 · 兩層復驗（內部快 + 外部慢）
- **脈絡**：分析可能建立在過期或錯誤的 DB 資料上。
- **決策**：快層比對 DB 內部一致性；慢層抽樣打 FinMind 外部複核。
- **取捨**：建立資料信任分級；換來 FinMind 額度限制下只能抽查、非全驗。

## ADR-10 · 單體而非微服務
- **脈絡**：單一開發者維運，無多團隊並行需求。
- **決策**：保持單一 codebase 單體，用模組邊界而非服務邊界切分。
- **取捨**：開發/部署極簡；換來無法獨立擴展單一組件、單點故障。對此系統規模是正確取捨。

## 決策相依關係
```
ADR-5(免費API) ─┬─► ADR-6(Decimal128) ─► 資料完整性
                └─► ADR-9(復驗)      ─┘
ADR-2(自架LLM) ─┬─► ADR-3(節點分工)
                └─► ADR-4(兩階段)   ─► ADR-8(雙寫)
ADR-1(共享DB) ──► ADR-10(單體) ──► ADR-7(cron批次)
```
