# 架構四面向詳解 · 1 · 架構風格 (Architecture Style)

> 反向工程自 `172.16.9.166` 實機。以軟體架構七大風格逐一評估符合度。

---

## 一句話定性

**分層模組化單體（主導）＋ 四個微核心（core+plugin）子架構（局部）＋ 封裝式與分散式插件並存。刻意不走微服務與事件驅動。**

---

## 七風格符合度矩陣

（● = 符合強度，5=主導、1=零星、0=不符）

| 風格 | 符合度 | 判定 | 體現位置 / 證據 |
|------|:--:|------|----------------|
| **分層架構** Layered | ●●●●● | 主導 | `src/` 五層封閉分層：取得→儲存→分析→決策→介面，逐層單向依賴 |
| **模組化單體** Modular Monolith | ●●●●● | 主導 | 單一 codebase/部署單元；15 個依領域切分模組（factors/indicators/morphology/moe…） |
| **微核心** Microkernel | ●●●●○ | 顯著·局部 | **4 個 core+plugin 子架構**（見下） |
| **封裝式插件** Encapsulated | ●●●●○ | 顯著 | 選股四法/回測策略/因子計算器＝同進程 import+介面呼叫 |
| **分散式插件** Distributed | ●●●○○ | 局部 | MoE LLM 角色＝`role_router` 經 HTTP 呼叫 .28/.27 Ollama |
| **微服務** Microservices | ●○○○○ | 不符 | 僅 API/Dashboard/Ollama 為獨立進程；業務未拆服務、**共享單一 DB**（反模式） |
| **事件驅動** Event-Driven | ○○○○○ | 不符 | 無 broker/事件匯流排/pub-sub；cron 時間觸發＝排程批次編排，非事件驅動 |

---

## 微核心子架構（四處 core + plugin）

系統反覆出現「核心定義流程與契約 + 可插拔插件實作」的模式。新增能力＝加插件、不動核心——這是可擴充性的來源。

| 微核心 | 核心 (core) | 插件契約 | 插件 | 連接型態 |
|--------|-------------|----------|------|:--:|
| **選股引擎** | `daily_recommendations.py` | `scan_*() → list[picks]` | 因子排行 · 蔡森 · 謝富旭 · 北大 | 封裝式 |
| **回測引擎** | `Backtest.run()` | `Strategy.generate_signals(date,data)` | MA交叉 · RSI回歸 · 價值動能 | 封裝式 |
| **MoE 團隊** | `role_router` / `team_daily_verified` | `ask_role(role,question)`；註冊表 `ROLE_TO_MODEL`+`ROLE_PROMPTS` | 7 角色 | 分散式 |
| **資料下載器** | `download_coordinator.py` | `DATA_TABLES` 配置 dict | 43 表配置 | 封裝式 |

> **為何是微核心而非只是模組化**：關鍵在**顯式插件註冊表 + 統一契約**。MoE 的 `ROLE_TO_MODEL`/`ROLE_PROMPTS`、下載器的 `DATA_TABLES` 都是 plugin registry——加角色/加表＝改註冊表，核心不變。

## 插件連接型態：封裝式 vs 分散式（本系統最有意思的特徵）

### 封裝式 Encapsulated（同進程、直接呼叫）
- **選股四法**：`daily_recommendations` import 並呼叫 `HsiehValueScreen`/`StockRanker`/…
- **回測策略**：`Backtest` 持有 `Strategy` 實例，呼叫 `generate_signals()`
- **因子計算器**：`FactorLibrary` 調度 momentum/quality/value/volume 四類
- **技術指標**：`indicators/*.py` 純函式吃 DataFrame
- 特徵：編譯期綁定、無網路、失敗即例外。優點快/簡單；限制不可獨立擴展、同語言。

### 分散式 Distributed（跨進程/跨機、over HTTP）
- **MoE LLM 角色**：`role_router.ask_role` → `POST {url}/api/generate` 到 .28/.27 Ollama
- **合議委員**：`consensus._ask` → .27 節點 HTTP
- **團隊取數**：`fetch_all_data` → 本地 API `:8888` 9 端點（而非直讀 DB）
- 特徵：執行期綁定、走 REST、需逾時/重試。優點可分攤多 GPU 節點、換模型不改核心；代價網路延遲、協調複雜。

> **架構師觀察**：把 LLM 角色設計成**分散式插件**是關鍵決策——讓「加一台 GPU 節點」或「換模型」變成設定變更（`MODEL_TO_URL`/`ROLE_TO_MODEL`），而非改程式。這也是 .27/.28 雙節點能無痛協作、以及 [2 節點試點瘦身版](規劃-2節點試點瘦身版.md) 能把兩節點併一台的原因。反之，量化選股/回測是封裝式——不需跨機，追求速度與簡單。

## 為何「不符」的兩個風格也是刻意選擇

- **不走微服務**：業務邏輯未拆服務、共享單一 MongoDB（微服務要求 per-service DB）。單人維運下，單體的簡單性遠勝分散式營運負擔（見 [架構決策 ADR-10](架構四面向詳解-3-架構決策.md)）。
- **不走事件驅動**：台股每日收盤結算，本質是批次；cron 排程編排足矣，引入 broker/事件匯流排是過度設計（見 ADR-7）。

## 與其他三面向的連結
- 微核心 + 插件直接服務 **[架構特性](架構四面向詳解-2-架構特性.md)** 的「可擴充性」。
- 分散式插件、共享 DB、cron 批次、單體 落實為 **[架構決策](架構四面向詳解-3-架構決策.md)** 的 ADR-2/1/7/10。
