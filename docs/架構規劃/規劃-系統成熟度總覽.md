# 規劃 · 系統成熟度總覽（被動告警 → 主動確保完整 + 可傳承）

> 一句話：這個系統從「**壞了才知道、只有一個人會修**」，演進到「**壞之前先擋、能自癒、修法寫下來人人可接手**」。
> 本文把散落各處的機制收束成三個維度，供未來的人／AI 一眼看懂系統走到哪、為什麼。

---

## 三個維度

資料系統的成熟度不是單一軸，是三軸同時長：

| 維度 | 從 | 到 | 白話 |
|---|---|---|---|
| **① 反應速度** | 被動告警 | 主動確保 | 壞了才叫 → 壞之前先擋、緩慢劣化先示警 |
| **② 修復能力** | 純人工 | 自癒＋升級 | 每次叫人 → 能修的自己修、修不了才叫人 |
| **③ 可傳承** | 口耳／單人 | 版控＋文件 | 只有一個人會 → 寫下來、進 git、誰都能接 |

---

## 各維度現況（具體元件）

### ① 反應速度：被動 → 主動（L0→L4）
- **L1 可信偵測**：完整度檢查排除 TAIEX（恢復停擺偵測）+ 時區守衛（`twse_openapi_sync.py`）
- **真相錨**：自適應門檻＝max(靜態, 近20日中位×0.85)，對照「近期常態」而非「跟昨天一致」
- **正確錨**：值合理性檢查（OHLC 自洽、均額）——驗「值對」不只驗「存在」
- **存活錨**：看門狗（`watchdog.py`）——確認排程 job 自己有在跑（誰看守看守者）
- **L3 主動驗證**：`data_health.py` 每日健康快照＋趨勢劣化預警、FinMind 外部交叉

### ② 修復能力：人工 → 自癒 → 升級
- **自癒**：`twse_openapi_sync.py --heal`（週一~五 21:00）check→自動補→複檢→仍失敗才升級人工
- **回補**：`backfill_by_date.py`（按日期補股價）、`backfill_recent_gaps.py`（隔日補 T+1 法人／PE，**分市場偵測半缺**）
- **升級**：修不了的（時區、pipeline 停擺、法人最新日半缺）發 🚨 給人，附 runbook 連結

### ③ 可傳承：口耳 → 版控＋文件
- **版控**：git + GitHub（`acronhuang/TW_STOCK`）——程式/設定不再是單機的部落知識
- **config-as-code**：`deploy/`（crontab + systemd + 災難復原 README）——機器毀了能重建
- **給 AI/人的導覽**：`llms.txt`（先讀地圖）、[檔案功能與關聯總目錄](規劃-檔案功能與關聯總目錄.md)（314 支程式逐檔）
- **運維知識**：[運維手冊](規劃-運維手冊-完整度異常處理.md)（收到告警怎麼辦，含每個踩過的坑）
- **決策可稽核**：合議雙定案並存（`blind_final` vs `final`）+ 討論逐字稿存 DB + `reverify_team` 復驗

## 一次真實事件的閉環（2026-07-14 法人半缺）

三個維度如何協作，用一次真實告警示範：

1. **① 偵測**：完整度檢查抓到 `institutional_flow 833 < 門檻 1743`（自適應門檻）
2. **② 自癒**：21:00 `--heal` 自動補好了 `stock_price`；但法人半缺補不了 → **升級人工**（心跳記 `need_human`）
3. **人工診斷**：發現是「有上櫃缺上市」，根因＝OpenAPI 端點慢 RWD 一天 → 用 `fetch_twse_institutional` 手動補 1226 筆
4. **治本**：修 `backfill_recent_gaps` 的半缺盲點（改分市場偵測）→ 未來此類自動補
5. **③ 可傳承**：把根因＋處置寫進 runbook「關鍵陷阱#5」+ 決策樹 → 下次照做即可

**這就是「被動告警 → 主動確保 + 可傳承」的完整閉環**：偵測 → 自癒 → 升級 → 診斷 → 治本 → 文件化。

## 誠實的殘留缺口

不粉飾——還沒做到滿的地方：

| 維度 | 殘留 |
|---|---|
| ① 反應 | 仍是輪詢式（cron），非即時；靜態門檻於全新部署的頭幾週樣本不足 |
| ② 修復 | `--heal` 補法人時抓昨日（T+1 預設），無法**當晚**自癒最新日半缺（靠隔日回補）|
| ③ 可傳承 | 本機 bundle 尚未改成 git clone（仍與 .166 靠 rput 同步）；無 CI（測試套件在，但未自動跑）|
| 安全 | API/Dashboard 無驗證、`.env` 明文（見 Backlog S1–S5）|

> 這些不是「沒想到」，是「已知、排優先序」——見 [紅線盤點與 Backlog](規劃-紅線元件盤點與Backlog.md)。**能誠實列出殘留，本身就是可傳承的一部分。**
