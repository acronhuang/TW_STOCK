# 規劃 · 支援設備硬體需求

> 現行三節點的硬體推估 + 若要重建/擴充的規格建議。部分為由負載反推之估算（標註）。

---

## 一、現行負載特徵

| 負載 | 特徵 | 瓶頸資源 |
|------|------|---------|
| 資料下載/寫入 | 每日批次，數千筆 upsert | 網路 I/O、Mongo 寫 |
| 因子/指標計算 | `parallel_factor_calculation --workers 4`，全市場 ~15min | CPU、Mongo 讀 |
| MongoDB | 5M+ stock_price、3.6M factors | RAM（工作集）、磁碟 |
| **LLM 推理（主）** | qwen2.5-14b 全層 GPU（num_gpu=99），全市場 team 分析 | **GPU VRAM + 算力** |

---

## 二、主機 `.166` 需求

| 資源 | 現況/建議 | 說明 |
|------|----------|------|
| CPU | 8 核+（估） | 因子平行計算 4 workers + API/Dashboard |
| RAM | 32GB+（估） | MongoDB 工作集（stock_price+factors 索引）+ Python |
| 磁碟 | 100GB+ SSD | 現用 24G/97G；DB + 日誌 + 備份成長 |
| 磁碟 I/O | SSD 必要 | stock_price 逐日查詢頻繁 |
| GPU | **不需** | 推理外包給 .28/.27 |

> ⚠️ MongoDB 8.0 需 **AVX** 指令集（實體 CPU 幾乎都有）。

## 三、推理節點 `.28` / `.27` 需求

模型 VRAM 佔用（q4 量化估算）：

| 模型 | 參數 | VRAM（估） |
|------|:--:|:--:|
| qwen2.5-14b | 14B | ~9–10 GB |
| qwen2.5-3b | 3B | ~2–3 GB |
| hermes3:8b | 8B | ~5–6 GB |
| llama-3-taiwan-8b | 8B | ~5–6 GB |

| 節點 | 需同時常駐 | 建議 VRAM |
|------|-----------|:--:|
| `.28` | qwen2.5-14b + qwen2.5-3b | **≥ 16 GB**（如 RTX 4080/A4000 級） |
| `.27` | hermes3:8b + qwen2.5-3b | **≥ 12 GB**（如 RTX 3080/4070 級） |

> `keep_alive=5m` 使模型常駐避免反覆冷載；VRAM 要能同時裝下該節點的常用模型。

## 四、若重建為「瘦身單機」（見 [2節點試點瘦身版](規劃-2節點試點瘦身版.md)）

| 方案 | 硬體 | 取捨 |
|------|------|------|
| 全合一單機 | 1 台含 1 張 ≥24GB GPU（如 RTX 4090 / A5000） | 省一台機；但 DB+推理搶資源，全市場更慢 |
| 主機 + 單 GPU 節點 | .166 + 1 台 ≥24GB GPU | 合議降級為「單節點多模型」，失去跨機多樣性 |

## 五、網路

- 三節點需在同網段（`172.16.9.x`），低延遲（LLM 呼叫頻繁）。
- 對外：僅需能連 TWSE/TPEX/FinMind/台銀（HTTPS out）。
- 若 Dashboard 要給人用：內網存取即可，**勿對公網開放 8501**（無驗證）。

## 六、容量成長預估

| 項目 | 日增 | 年增（估） |
|------|:--:|:--:|
| stock_price | ~6000 筆 | ~1.5M 筆 |
| stock_factors | ~2000 筆 | ~0.5M 筆 |
| team_analysis | ~2000 筆/次全跑 | 視頻率 |
| 日誌 | 數十 MB | log_rotation 每日清 >30 天 |

→ 磁碟年增約數十 GB，100GB SSD 可撐 1–2 年，屆時擴充或歸檔舊資料。
