# 規劃 · 推薦模組選型

> 針對現有技術棧，逐層給「保留 / 替換 / 補強」建議。以「單人維運、資料正確優先、低成本」為選型準則（承 [架構特性](架構四面向詳解-2-架構特性.md)）。

---

## 一、資料庫

| 現況 | 評估 | 建議 |
|------|------|------|
| MongoDB 單機 | 文件型契合多源異構資料；但 schema 無約束是紅線根源 | **保留 MongoDB**；補 **JSON Schema validation**（`collMod` 加 `$jsonSchema`）於核心表，把隱性契約顯性化 |
| 無時序優化 | stock_price 5M 筆按日查詢 | 現有 `(symbol,date)` 索引已足；量再大再議 time-series collection |

## 二、LLM 推理

| 角色 | 現用 | 建議 |
|------|------|------|
| 主力推理（技術/基本面/顧問） | qwen2.5-14b | **保留**；14b 對繁中財經足夠 |
| 輕角色（總經/風險） | qwen2.5-3b | 可保留；若品質不足升 7b |
| 合議委員 | hermes3:8b + qwen2.5-3b | **升級 3b→7b**，或改 `qwen2.5-7b` + `gemma2-9b` 增家族多樣性 |
| 繁中潤稿 | llama-3-taiwan-8b | 保留（在地化強） |

> 選型準則：全部維持**開源自架**（成本 ADR-2 不變）。避免引入需付費 API 的模型。

## 三、API 層

| 現況 | 建議 |
|------|------|
| FastAPI，無驗證，綁 localhost | **保留 FastAPI**；補 **API Key 中介層**（單一 header token 即可，內網夠用）；若要對外，前置 nginx + TLS |

## 四、儀表板

| 現況 | 建議 |
|------|------|
| Streamlit 1.59，systemd | **保留 Streamlit**（單人開發最快）；重運算頁一律用 `@st.cache_data`；大清單一律文字輸入（已落實 charts） |

## 五、排程

| 現況 | 建議 |
|------|------|
| cron（時區已修） | **保留 cron**；補一支 **healthcheck 心跳**（每日檢查關鍵服務 + 資料新鮮度 → LINE），把「可觀測性」從 ② 拉到 ③ |

## 六、資料驗證/復驗

| 現況 | 建議 |
|------|------|
| completeness check + 兩層復驗（新增） | **保留並常態化**；把 `reverify_team` 快層納入每日 cron，慢層每週抽查 |

## 七、版控/部署（最弱環節）

| 現況 | 建議 |
|------|------|
| 無 CI/CD，手動部署，本機/正式機分歧 | **導入最小 CI**：git + 一支 `deploy.sh`（rsync 到 .166 + restart systemd + smoke test）；先把 `.166` 變更 pull 回 git 消除分歧 |

---

## 選型優先序

1. **P0**：git 化 + 消除本機/正式機分歧（無版控是最大風險）。
2. **P1**：核心表加 JSON Schema validation（治本 schema 紅線）。
3. **P1**：API Key + healthcheck 心跳。
4. **P2**：合議委員模型升級（見 [單一vs合議](規劃-研判模型策略-單一vs合議.md)）。
5. **P2**：最小 CI/deploy 腳本。

> 原則：**不換技術棧，只補約束與自動化**。現有選型對此系統規模都是對的，痛點在「缺約束、缺自動化」，非「選錯工具」。
