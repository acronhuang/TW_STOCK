## ⚡ 先讀：現在系統會「自癒」，多數異常不用你動手

**2026-07 起，21:00 的 cron 已從 `--check-only` 升級為 `--heal`**——它會自動嘗試修復，只有修不好才升級要你介入。你收到的 LINE 有三種：

| LINE 訊息 | 意思 | 你要做的 |
|---|---|---|
| `✅ 全部正常` | 檢查通過 | 無 |
| `🔧 已自動修復 ✅` | 有異常但自癒成功 | 無（知道就好）|
| `🚨 自癒後仍…需人工介入` | 自癒補不了 | **照下方手動流程處理** |

另有主動監控（P2）會在完整度還沒跨門檻前先示警：
| LINE 訊息 | 意思 |
|---|---|
| `📉 覆蓋率緩慢劣化` | 近 7 天覆蓋率下降，趨勢向下（`data_health --trend`）|
| `🚨 FinMind 交叉驗證符合率低` | DB 收盤與外部源背離（`data_health --finmind`）|
| `❌ 系統時區異常` | 時區漂移（守衛觸發）→ **立即查，這是最危險的根因** |
| `🐕 看門狗：排程未如期執行` | **檢查/健康 job 自己沒跑**（cron 死/腳本 crash/時區）→ 見第 3 步查 cron |

> **看門狗（C · 存活錨）**：整套防護都假設 21:00/22:00 的 job「會跑」。每個 job 完成時寫一枚心跳（`system_heartbeat` 集合）；獨立的 `watchdog.py`（每天 08:00、14:00）比對「上次執行是否落在應執行的排程窗內」，逾期就發 `🐕`。這補上「誰看守看守者」的盲點——job 靜默停擺不再無人知曉。收到 `🐕` 幾乎必是 cron/時區問題，直接跳第 3 步。

**所以：收到 `🔧 已修復` 只需知悉；收到 `🚨` 或 `📉` 或時區告警才照下方 runbook 處理。**

### 自癒能修什麼、不能修什麼
| 可自癒（自動補資料）| 不自癒（升級人工，系統級根因）|
|---|---|

| stock_factors（重算因子）| 時區異常（改系統設定太危險）|
| 11 張補充表（重跑 sync）| |
| institutional_flow（重抓法人）| |

## 背景：檢查在檢查什麼

`check_integrity(db)`（`twse_openapi_sync.py`）對 13 張表逐一驗三件事：
1. **最新日筆數 ≥ 門檻**（抓「當天只收一半」）
2. **落後 ≤ 容許交易日數**（抓「整天漏掉」）
3. **值合理性**（僅 stock_price）：OHLC 是否自洽（抓「筆數夠但價格全錯」）

參考交易日 = `stock_price` 最新 `date`（**已排除 TAIEX**，P0 修正——否則 macro_sync 每日塞的 TAIEX 會讓停擺偵測失效）；落後以**交易日**計（近 120 天 stock_price 非 TAIEX 出現過的日期為交易日曆）。另有 `PIPELINE_STALE_DAYS=5`：stock_price 落後今日 >5 **日曆天** → 判 pipeline 停擺。並在檢查開頭做**時區守衛**（`/etc/localtime` 須為 Asia/Taipei）。

### 自適應門檻（B/D · 真相錨點）
「筆數門檻」不再只看寫死的靜態值——那是脆弱的（設太低會漏抓、設太高會誤報）。對**固定母體、每交易日筆數穩定**的表（`_ADAPTIVE_TABLES`：股價/因子/法人/融資券/當沖/借券/盤後/零股/外資 Top20/ETF），門檻 = `max(靜態地板, 近20交易日中位數 × 0.85)`：
- **只往上抬，不往下降**：靜態仍是絕對地板，中位數把門檻抬到「近期常態」。修正「門檻太低→漏抓」——例：`after_hours_trading` 靜態只設 1000，但真實日量 ~15000，舊門檻讓它掉到 1100 也過關；新門檻要求 ~12958。
- **爆發/稀疏表不套用**（重大訊息/處置股/停資停券預告）：筆數天生大起大落，中位數當門檻會誤報，維持靜態低門檻 + 容許落後天數即可。
- 中位數即「真相錨點」：拿系統自己近 20 天的常態當基準，而非跟昨天單點比。報告會印 `（門檻X｜中位Y）`。

### 值合理性檢查（正確錨 A）
筆數/落後只驗「存在」，不驗「值對」——「筆數夠但收盤全錯」的日子會過關。`_value_sanity_bad` 對 stock_price 當日驗**零成本內部不變量**（不打外部 API）：有成交(volume>0)時 OHLC 須 >0；`low≤high`；close/open 落在 `[low,high]`；volume≥0。這些是結構性、正常解析不可能違反的。逾 `_VALUE_BAD_MAX=5` 筆 → 併入 `stock_price` 異常行 → 走同一條 backfill 自癒（重抓可修來源暫時錯批；系統性 bug 則複檢仍紅→升級人工）。正常時報告印 `值檢✓`。
> 校準：未成交股（volume==0，OHLC 皆 0）是合法「當日無交易」，故 >0 檢查僅對 volume>0 生效，不會誤報。

### 第 2 步 · 依類型補正

| 異常 | 受影響表 | 處理指令 |
|------|---------|---------|
| 筆數不足 | stock_price / institutional_flow | `.venv/bin/python3 scripts/twse_daily_update.py`（不帶參數＝抓 API 最新可得日） |
| 筆數不足 | stock_factors 技術因子 | `.venv/bin/python3 scripts/parallel_factor_calculation.py --workers 4 --start-date X --end-date Y` |
| 筆數不足 | 補充表 | `.venv/bin/python3 scripts/twse_openapi_sync.py --only <表名>` |
| 落後 1 交易日 | institutional_flow / stock_factors PE | 多為 **T+1 延遲**，隔日 `daily_senvision.sh` 的 `backfill_recent_gaps` 自動補；急則手動重跑 twse_daily_update |
| 過去某日缺（API 已跳過） | stock_price | **按日期端點**：改 `backfill_missing_dates.py` 日期範圍（MI_INDEX），或 TPEX 新站台 `www.tpex.org.tw/www/zh-tw/afterTrading/otc?date=YYYY/MM/DD&type=EW&response=json` |
| pipeline 停擺 >5 天 | 全部 | 先查 cron（見第 3 步） |

### 第 3 步 · pipeline 停擺排查（最常見根因）
```bash
timedatectl                              # 時區必須 Asia/Taipei（非 UTC！）
systemctl is-active cron                 # 須 active
tail -20 logs/cron_daily_senvision.log   # 每日主更新最後執行與錯誤
journalctl -u cron --since "2 days ago" | grep senvision   # cron 是否真觸發
```
> ⚠️ **歷史根因**：系統曾停在 UTC，cron 用 UTC 判讀 → 每日更新實跑台北凌晨 01:00 且少抓一天。時區已修（`timedatectl set-timezone Asia/Taipei`），但此為「停擺」類最常見元兇。

### 第 4 步 · 驗證收尾
```bash
.venv/bin/python3 scripts/twse_openapi_sync.py --check-only     # 應全綠
# 若補的是團隊分析要用的資料，順手復驗
.venv/bin/python3 scripts/reverify_team.py --date YYYYMMDD --finmind 30
```

## 關鍵陷阱

1. **`twse_daily_update.py --date` 不能抓過去日期**。它打 OpenAPI「最新交易日」端點，`--date` 只是把抓回的資料**事後過濾**。補過去某天須用**按日期查詢**端點（MI_INDEX / TPEX 新站台 `otc?date=`）。

2. **TAIEX 汙染參考日**。`macro_sync` 每日往 stock_price 塞一筆 `symbol='TAIEX'`。若某天只有這一筆，`筆數 1 < 4000` 會觸發，且它會讓其他表的「落後參考日」虛報一天。判斷 `stock_price` 當天是不是只剩 TAIEX：`distinct('stock_id',{date:d})` 若只回 `["TAIEX"]` 即是。

3. **免費 API 天生缺口**：冷門股/權證/下市股在免費 OpenAPI 常缺，FinMind 也未必有。這類「少數檔缺」屬正常，非 pipeline 問題（見上次 45 檔 unknown 皆為停牌/下市股）。

4. **T+1 延遲**：三大法人(T86)、PE/PB(BWIBBU) 當日 17:00 常尚未公布，`institutional_flow` 容許落後 1 交易日即為此設計。隔日回補為正常流程，非異常。

## 快速決策樹

```
LINE 告警
 ├─ "pipeline 停擺 >5天"  → 查 cron/時區（第3步）→ 修好後手動補缺天
 ├─ "筆數 X < 預期"
 │   ├─ 當天只剩 TAIEX 1 筆 → 每日更新沒跑成 → 重跑 twse_daily_update
 │   ├─ 只缺 TPEX 或 TWSE 一半 → 重跑 twse_daily_update（或按日期端點補過去日）
 │   └─ 補充表 → twse_openapi_sync --only <表>
 └─ "落後 N 交易日"
     ├─ N=1 且為法人/PE → T+1 正常，等隔日自動補
     └─ 因子落後 → parallel_factor_calculation 重算該窗口
最後一律：twse_openapi_sync --check-only 複驗全綠
```
