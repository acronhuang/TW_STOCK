# 規劃 · 決策議程與 RACI

> **假設**：本系統目前為單人（Ming）開發維運。以下 RACI 以「若擴為小團隊」的角色抽象填寫；單人情境下所有角色由同一人兼任，本表用途轉為「決策檢查清單」。角色請依實際組織調整。

---

## 角色定義（抽象）

| 代號 | 角色 | 職責 |
|------|------|------|
| **OWN** | 系統負責人 | 最終決策、優先序 |
| **ARCH** | 架構/技術負責人 | 技術方案、選型 |
| **DATA** | 資料/量化 | 資料正確性、策略邏輯 |
| **OPS** | 維運 | 部署、排程、監控 |

> R=負責執行 · A=最終當責 · C=諮詢 · I=知會

## 決策會議建議議程（若召開）

1. **回顧**（10min）：架構四面向摘要 + 本次修正清單。
2. **P0 決策**（20min）：D1 安全暴露、D5/D6 版控與分歧、D8 風控失效。
3. **P1 方向**（20min）：D2 合議策略、D3 模型、D4 財報統一。
4. **資源**（10min）：D9 硬體/試點。
5. **行動項**：每項指派 R + 期限。

## 單人情境的簡化用法
既為單人維運，本表實際用途是**「決策前自我檢查清單」**：每做一個架構變更，問自己——
- 這是否改變某個 ADR？→ 更新 [架構決策](架構四面向詳解-3-架構決策.md)。
- 這是否動到 DB schema（隱性契約）？→ 檢查下游、更新 [邏輯組件](架構四面向詳解-4-邏輯組件.md)。
- 這是否引入新紅線或還了技術債？→ 更新 [Backlog](規劃-紅線元件盤點與Backlog.md)。
