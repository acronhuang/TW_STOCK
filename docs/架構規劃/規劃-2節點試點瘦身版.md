# 規劃 · 2 節點試點瘦身版

> **假設**：目標是在最少硬體（2 節點）上，跑一個「可驗證核心價值」的瘦身試點，用來驗證系統在新環境可運作、再決定是否全量部署。若你的目標不同（例如原地強化），請告知調整。

---

## 試點目標（可驗證）

在 2 台機器上，於 1 週內證明：
1. 資料管線能自動抓齊台股每日行情與因子。
2. 四法選股能每日產出推薦並推 LINE。
3. 團隊分析能對「當日推薦清單」（非全市場）產出 7 角色 + 合議定案。
4. 儀表板能瀏覽上述結果。

**刻意排除**（非試點必要）：全市場 2000 檔團隊分析、衍生性商品/黃金/期權等 43 表全量、歷史回填多年。

```
┌────────────────────────────┐     ┌──────────────────────────┐
│ Node-A（主機）              │     │ Node-B（推理）            │
│ Ubuntu                     │     │ 1× GPU ≥16GB             │
│ - MongoDB                  │◄───►│ - Ollama                 │
│ - FastAPI :8888            │ LAN │   qwen2.5-14b            │
│ - Streamlit :8501          │     │   qwen2.5-3b            │
│ - cron + 全部 Python       │     │   hermes3:8b（合議同機） │
```

**瘦身取捨**：`.27` 與 `.28` 合併為單一 Node-B。合議委員與主力推理跑同機不同模型 → 失去「跨機模型家族多樣性」，但保留「多模型投票」。設定：`OLLAMA_URL = OLLAMA_CONSENSUS_URL = http://Node-B:11434`。

## 部署步驟（對照 [部署Prompt手冊](規劃-部署Prompt手冊.md)）

1. Node-A：裝 MongoDB 8.0 + Python 3.12 venv + 專案。
2. `mongorestore` 近 2 年資料（或從頭抓 30 天暖機）。
3. Node-B：裝 Ollama + pull 3 模型。
4. `.env` 指向 Node-B（兩個 OLLAMA URL 同一台）。
5. 建 systemd（api + dashboard）+ cron（daily_senvision + daily_recommendations + team_daily_50）。
6. Smoke test：`/api/health`、跑一次 `daily_recommendations --no-line`、跑 `team_daily_verified --symbols 2330`。

## 風險與回退
- **單 GPU 記憶體不足**：3 模型同時常駐需 ≥16GB；不足則減 `keep_alive` 或減委員模型（合議降為單委員 + 顧問草案）。
- **回退**：試點與正式環境資料庫分離，試點失敗不影響 `.166` 現行運作。
